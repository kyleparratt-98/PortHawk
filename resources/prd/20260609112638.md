## Programming Language

Python

## Original Requirements

Build a Python CLI tool that monitors local TCP/UDP port activity in real-time, extracts connection states and process names per port, and visually outputs them to standard out using a live-updating bordered table with color-coded status indicators. It must handle permission-denied errors for privileged ports gracefully.

## Project Name

port_sentry

## Product Goals

- Deliver a real-time, intuitive, and visually clear overview of local network port activity for developers and system administrators.
- Ensure robust operation by gracefully handling system-level errors and permission constraints.
- Provide actionable insights by correlating ports with processes and connection states efficiently.

## User Stories

- As a backend developer, I want to see which ports my application is listening on and their connection states, so I can debug network issues quickly.
- As a system administrator, I need to monitor all active TCP/UDP connections in real-time with process names, so I can identify unauthorized or suspicious activity.
- As a DevOps engineer, I want a lightweight CLI tool that runs without complex setup and handles permission errors without crashing, so I can use it in various environments.

## Competitive Analysis

- netstat (command-line): Built-in system tool, provides raw data but not real-time updating, requires parsing, no color-coding or process names by default on all systems.
- lsof (command-line): Powerful for listing open files/ports, but output is verbose, not designed for real-time monitoring, and requires specific flags for port filtering.
- nmap: Excellent for network discovery and security scanning, but not intended for continuous, real-time monitoring of local host ports.
- Windows Resource Monitor (GUI): Provides real-time network tab with processes, but is GUI-only, Windows-specific, and lacks a concise CLI table output.
- iftop / nethogs: Real-time network bandwidth monitors by interface/process, but do not focus on per-port connection states (LISTEN, ESTABLISHED, etc.).
- ss (command-line): Modern replacement for netstat, faster, but like netstat, it's a snapshot tool without a live, formatted, color-coded table interface.
- bmon/iptraf: Console-based network monitoring, more focused on bandwidth and traffic rather than a simple, clean table of port->process->state.

## Requirement Analysis

The core requirement is a real-time monitoring CLI tool. Key technical aspects include:
1. **Data Source**: Need to poll or listen for system network connection data (e.g., via `/proc/net/tcp`, `/proc/net/udp`, `psutil` library, or `ss`/`netstat` command output). The `psutil` library in Python is cross-platform and provides process information, making it a strong candidate.
2. **Real-time Output**: Requires a live-updating console display. Libraries like `rich` or `curses` can create formatted, bordered tables that update in-place.
3. **Visual Design**: Must include a bordered table, color-coded status indicators (e.g., green for LISTEN, yellow for ESTABLISHED, red for TIME_WAIT).
4. **Data Points per Row**: Protocol (TCP/UDP), Local Address:Port, State (for TCP), Process Name/PID, possibly Remote Address.
5. **Error Handling**: Must catch `PermissionError` or similar when accessing privileged ports (e.g., ports < 1024) and display a placeholder (e.g., '[root required]') instead of crashing.
6. **Performance**: Should update at a reasonable interval (e.g., every 1-2 seconds) without high CPU usage.
7. **CLI Interface**: Should accept basic arguments like refresh interval, filter by protocol, or port number.

## UI Design draft

A full-screen terminal UI with a centered, bordered table. Table columns: Proto, Local Address, State, PID, Process Name. The table header is bold. Row colors: LISTEN (green), ESTABLISHED (yellow), TIME_WAIT (red), others (default white). The bottom of the screen shows a status bar with instructions: 'Port Sentry | Q:Quit, P:Pause, R:Refresh'. If a permission error occurs, the Process Name cell shows '[Permission Denied]' in dim red. The entire display refreshes smoothly every second.

## Anything UNCLEAR

The requirement mentions 'UDP port activity' and 'connection states'. UDP is connectionless, so typical states (LISTEN, ESTABLISHED) don't apply. Clarification: Should UDP ports only show 'LISTEN' (if a process has the UDP port open) or also show active datagram traffic? The assumption for the initial implementation will be to show UDP ports in a 'LISTEN' state when bound by a process.
